Research Findings

Three Key Insights

These aren't prescriptions — they're observations from someone who spent time actually using the product and reading the docs as a developer would.

1

Navigation is organized around product modules, not developer goals

The sidebar reflects internal structure — Entitlements, Paywalls, Experiments — rather than what a developer is trying to accomplish. When thinking "I want subscriptions working," I had to figure out which module mapped to that intent. A goal-first IA would surface Getting Started → Implementation → Monetization instead.

2

The conceptual layer is missing — developers jump in cold

There's no single place explaining what the core objects are, why they exist, and how they relate before you start coding. I had to reconstruct the relationship between Products, Offerings, Packages, and Entitlements across several pages. A "Core Concepts" section with diagrams would orient new developers in minutes instead of hours.

3

API docs are informational, not interactive

Endpoints are documented as static text. There's no try-it-out console, no schema-driven examples, no way to test authentication inline. OpenAPI as the source of truth — rendered with something like Scalar — would give developers an interactive, Stripe-like experience without leaving the docs.
