Information Architecture

Hypothesis: Reorganizing for the Developer Lifecycle

The current "Dashboard vs SDK" split is a feature-based taxonomy. I propose shifting to a lifecycle-based taxonomy to reduce initial cognitive load:

Foundation

Unified quickstarts combining Dashboard and SDK setup into a single "Day 1" flow.

Conversion

Grouping Paywalls, Offerings, and Web Billing—the tools for getting developers paid.

Optimization

Focusing on Experiments and Targeting for apps ready to scale their revenue.

Operations

Clear paths for troubleshooting, customer center management, and data exports.

System Efficiency

Bridging the Engineering-to-Docs Gap

Accuracy is about process, not just prose. I'd focus on building the "maintenance loop" to ensure docs never lag behind code:

  • • Definition of Done: Integrating doc requirements into PR templates.
  • • Sync Loops: Pulling code snippets directly from SDK sample apps to ensure 100% accuracy.
  • • AI Readiness: Implementing llms.txt to feed the latest SDK context to AI-first developers.
