Research

What I noticed
while exploring RevenueCat

These aren't prescriptions ‚Äî they're observations from someone who spent time actually using the product and reading the docs as a developer would. I flagged what felt like friction points and thought about how I'd approach them as a manager.

Context: I read through the current RevenueCat docs, attempted a hands-on integration plan for a cross-platform app, and compared the experience against what I'd expect from developer documentation at this level. Everything below is based on that process ‚Äî not guesswork.

#1

Navigation is organized around product modules, not developer goals

The sidebar reflects internal structure ‚Äî Entitlements, Paywalls, Experiments ‚Äî rather than what a developer is actually trying to accomplish. When I was thinking "I want subscriptions working," I had to figure out which module mapped to that.

Each top-level section maps to a user intent, not an internal feature name. The concepts still live there ‚Äî they're just surfaced in context.

#2

The conceptual layer is missing ‚Äî developers jump in cold

There's no single place that explains what the core objects are, why they exist, and how they relate to each other before you start coding. I had to piece together the relationship between Products, Offerings, Packages, and Entitlements across several pages.

A "Core Concepts" section with diagrams and relationship maps would get new developers oriented in minutes instead of hours.

#3

Onboarding measures "steps completed," not "outcome achieved"

The quickstart says "create a project" and "add a product" ‚Äî but there's no clear north star like "make your first test purchase and see it in the dashboard." Outcome-based guides would give developers a finish line to aim for.

What verification looks like

After each guide, a "How to verify" block: make a sandbox purchase ‚Üí confirm entitlement is active ‚Üí see it reflected in the dashboard. It takes 30 seconds and builds confidence that the integration is actually working.

#4

API docs are informational, not interactive

Endpoints are documented as static text. There's no try-it-out console, no schema-driven examples, no way to test authentication inline. For a product whose core value is simplifying payments, the API experience could match that standard.

Potential approach

OpenAPI as the source of truth, rendered with something like Scalar ‚Äî giving you a try-it-out console, copyable examples, and schema validation without leaving the docs. This is a bigger lift, but it's what separates good dev docs from great ones.

‚Üí

How I'd phase this (if it were my call)

I wouldn't try to do everything at once. Here's the order that makes sense to me based on what had the most friction:

Phase 1

Core Concepts + Diagrams

Write the foundational "what is this and why does it exist" pages. Add relationship diagrams. This unblocks everything else.

Phase 2

Goal-Based Navigation

Reorganize the sidebar around user intent. Add cross-links between related concepts, guides, and API endpoints.

Phase 3

Verification Loops

Add "how to verify" blocks after every guide. Add outcome-based learning paths.

Phase 4

Interactive API Docs

OpenAPI spec + interactive renderer. Bigger investment, but high payoff for developer experience.
