Research

What I discovered
about RevenueCat

These aren't prescriptions ‚Äî they're observations from deep research into how developers actually use the product, where they get stuck, and what causes them to abandon integrations. I focused on the "vibe coder" persona: indie developers and small teams who want to experiment quickly and iterate.

My methodology: I conducted extensive research using Google to explore RevenueCat's SDK documentation site, user reviews, Reddit discussions, and developer sentiment across forums. I synthesized findings in NotebookLM and used Claude/Gemini to help structure and communicate the insights below. The analysis and strategic recommendations are mine.

#1
üî• Critical

Too much prerequisite setup before code

The documentation forces developers through extensive dashboard configuration (projects, products, entitlements, offerings) before they can write a single line of code. Vibe coders want to start coding immediately and see results.

What developers are saying

"I just wanted to try it out and spent 45 minutes in the dashboard before I even opened my IDE." ‚Äî Reddit user

Quick win

The "Test Store" feature exists but it's buried. Make it the hero of the quickstart: "Skip the dashboard setup ‚Äî use Test Store to start coding in 60 seconds."

#2
üî• Critical

Platform-specific API keys are confusing

RevenueCat uses separate API keys for iOS and Android (starting with app... and goog...). Multiple developers thought "app" meant "application" not "Apple." One developer wasted 3 weeks on this.

The problem

Most SDKs use a single API key. This multi-key system is non-standard and poorly explained upfront.

Solutions

  • Add a prominent callout box in the quickstart explaining the key naming convention
  • Include a visual diagram showing which key goes where
  • Add validation in the SDK that gives a helpful error if you use the wrong key
#3
‚ö†Ô∏è High

Error messages are cryptic

Error messages like "Error 23: There is an issue with your configuration. Check the underlying error for more details" provide no actionable information. For vibe coders who want to iterate quickly, cryptic errors are productivity killers.

Better error message

Error 23: Your iOS API key (app_xxx) doesn't match this project. Double-check you're using the key from the same RevenueCat project. Common fix: Verify in Dashboard ‚Üí App Settings ‚Üí API Keys

#4
‚ö†Ô∏è High

No simple "Hello World" example

There's no minimal, copy-paste example that shows a complete working flow from configuration to purchase in 10 lines of code. The quickstart jumps into comprehensive setup rather than showing the simplest possible working example first.

What vibe coders want

A npx create-revenuecat-app command that scaffolds a working project with Test Store configured, ready to run immediately.

#5
üìä Medium

Too much theory before practice

The docs explain concepts like "entitlements vs products vs offerings" extensively before showing working code. Vibe coders prefer learning by doing ‚Äî show the code first, explain the architecture later.

Proposed structure

  • Path 1: Fast Start ‚Äî Working code in 5 minutes, minimal explanation
  • Path 2: Deep Dive ‚Äî Comprehensive concepts and architecture
#6
‚ö†Ô∏è High

Configuration troubleshooting is reactive, not proactive

The most common issue (empty products/offerings) has an entire troubleshooting page, but this information should be front-loaded in the setup guide with clear warnings about common pitfalls.

Solution

Add "üö® Common mistake" callouts directly in the setup steps where errors typically occur, not buried in a separate troubleshooting section.

#7
üìä Medium

Cross-platform product configuration is unclear

Setting up products that work on both iOS and Android is confusing. The docs don't clearly explain how to link products across platforms or that you need to create platform-specific products in both stores.

#8
üìä Medium

Sandbox vs Production path is murky

The transition from Test Store ‚Üí Sandbox ‚Üí Production isn't clearly laid out as a progression. Developers struggle understanding when to connect real stores versus using the Test Store.

Needed

A clear visual roadmap showing the three stages with when/why/how to move between them.

What would help vibe coders

  • One-command starter: npx create-revenuecat-app that sets up a working project with Test Store
  • Minimal working example first: 10 lines of code to show a paywall and complete a purchase
  • Better error messages: Include likely causes and fixes directly in the error text
  • "Fast path" vs "Complete path": Separate docs for quick/dirty starts vs. production-ready implementations
  • Interactive configuration checker: A tool that validates your setup and highlights misconfigurations
‚Üí

How this maps to the role

The friction points above directly address the role's core challenges:

Information Architecture ‚Üí Findings #1, #2, #5

Current IA is product-centric, not user-goal-centric. My "Fast Path vs Complete Path" proposal directly addresses serving different user segments (new apps vs existing apps vs partial feature users).

Preventing Documentation Drift ‚Üí Finding #6

Proactive troubleshooting embedded in setup docs prevents the "docs falling out of date" problem. This is a process design fix, not just content cleanup.

AI Readability ‚Üí Findings #3, #4

Structured error messages and minimal working examples make docs machine-parseable. LLMs can generate better code if examples are complete, isolated, and copy-paste ready.

Cross-Team Coordination ‚Üí Finding #7

Cross-platform confusion reveals gaps in how Engineering, Product, and Docs coordinate. This needs a process fix: pre-ship documentation reviews and platform-specific validation checklists.

Information Architecture

The Knowledge Taxonomy

To scale documentation in an async-first environment, we must move beyond a "flat" list of pages. This taxonomy transforms our documentation into a structured data set that serves both developers and AI agents.

1. The Object Hierarchy

We map RevenueCat's technical entities to a clear four-tier taxonomy. This ensures that every page has a "home" and developers always know their location in the integration lifecycle.

Domain Entities Primary Goal
Product Catalog Products, Packages, Offerings, Entitlements Defining the "Source of Truth" configuration.
Implementation SDKs, User IDs, Aliasing, Testing Writing code and handling the purchase lifecycle.
Growth Paywalls, Experiments, Targeting Optimizing revenue via remote configuration.
Operations Webhooks, Analytics, Customer Lists Post-purchase data flow and business logic.

2. The Metadata Schema

By treating "Docs-as-Code," we tag every page with frontmatter metadata. This allows us to build a "Self-Service" KB where users can filter by their specific tech stack or business role.

---
rc_entity: Offering
platform: [iOS, Android, Stripe]
audience: [Developer, Product Manager]
lifecycle: General Availability
api_version: v2
---

Strategic Value

  • AI Readiness: Provides context for RAG systems.
  • Contextual Filtering: Allows a Marketer to see "Offerings" without the "API v2" technical specs.
  • Maintenance: Easily identify pages that need updates when a specific SDK version is deprecated.

3. The "Mapping Guide" Concept

One of my Month 1 goals is to establish a "Mapping Guide" within the docs. This bridges the gap between store SKUs and code-side logic, which is currently a major point of developer friction.

Pragmatic Benefit

Instead of developers searching through three different Apple/Google docs, our KB provides a single table mapping their Store Product IDs ‚Üí RC Packages ‚Üí Internal Entitlements. This reduces support tickets by providing "Ground Truth" documentation.

The AI Strategy: "Hub & Spoke"

This taxonomy enables a "Hub & Spoke" model. The documentation becomes the central brain (Hub) that powers various AI and human endpoints (Spokes).

The Hub (Source)

Structured, metadata-rich Markdown in Docusaurus. The single source of truth.

Spoke 1: The Human Reader

The standard documentation site, optimized for skimmability.

Spoke 2: The Agentic Layer

RAG-based support agent (Kapa/Algolia) for "How do I..." questions.

Spoke 3: The IDE Context

Exposing llms.txt so Cursor/Copilot can autocomplete valid code.
